# Beeber Boards & Cards

AI-powered platform for designing, balancing, and producing printable board and card games from natural language descriptions.

## Project Structure

```
app/              → React/Vite/Tailwind frontend
  src/
    App.tsx       → Root layout, project persistence, sidebar nav, stage routing
    api.ts        → Typed API client for all backend endpoints
    types.ts      → Core types: AppState, Stage, Card, CardType, BriefSettings, SimulationData
    constants.ts  → Design system tokens, stage definitions
    components/
      UI.tsx      → Shared components: Button, Card, Badge, cn() utility
    stages/       → One component per pipeline stage
      BriefStage.tsx    → Stage 1: Natural language input + AI concept generation (wired to API)
      DesignStage.tsx   → Stage 2: AI card type/instance generation + data grid
      BalanceStage.tsx  → Stage 3: Simulation dashboard (wired to API)
      ArtStage.tsx      → Stage 4: Style guide + asset gallery (placeholder)
      LayoutStage.tsx   → Stage 5: Canvas card editor (placeholder)
      ExportStage.tsx   → Stage 6: Print-and-Play PDF export (functional)
backend/          → FastAPI backend (Python 3.11+)
  app/
    main.py       → FastAPI app entry point, CORS, lifespan
    config.py     → Settings via pydantic-settings (.env loading)
    database.py   → Async SQLAlchemy engine + session factory
    models/
      project.py  → SQLAlchemy models: Project, GameConcept, CardType, Card
    schemas/
      project.py  → Pydantic schemas for request/response validation
    routers/
      projects.py → CRUD endpoints: projects, card types, cards
      generate.py → AI generation: concepts, card types, cards, simulation
      export.py   → PDF export endpoint
    services/
      concept_generator.py → Claude API concept generation with mock fallback
      card_generator.py    → Claude API card type/instance generation with mock fallback
      pdf_export.py        → Print-and-Play PDF generation via reportlab
  tests/          → pytest test suite (33 tests)
research/         → Market research, personas, competitive analysis, PRD, implementation plan
```

## Tech Stack

### Frontend
- **Framework**: React 19 + TypeScript 5.8
- **Build**: Vite 6
- **Styling**: Tailwind CSS v4 (uses `@theme` directive, NOT `tailwind.config.js`)
- **Animation**: Motion (Framer Motion v12+, imported from `motion/react`)
- **Charts**: Recharts 3
- **Icons**: Lucide React
- **Utilities**: clsx + tailwind-merge (combined in `cn()` helper in UI.tsx)

### Backend
- **Framework**: FastAPI (Python 3.11+)
- **Database**: SQLAlchemy 2.0 async + aiosqlite (SQLite for dev, Postgres for prod)
- **Validation**: Pydantic v2 + pydantic-settings
- **AI**: Anthropic Claude API (claude-sonnet-4-20250514) with mock fallback
- **PDF**: reportlab 4.x for Print-and-Play generation
- **HTTP client**: httpx (for external API calls)
- **Testing**: pytest + pytest-asyncio + httpx (ASGI transport)

## Key Conventions

### Tailwind v4
This project uses Tailwind CSS v4. Theme customization is done via `@theme` in `index.css`, not in a config file. Custom colors are CSS custom properties:
- `--color-bg`, `--color-surface`, `--color-surface-light`
- `--color-accent`, `--color-accent-hover`, `--color-ai`
- Use as Tailwind classes: `bg-bg`, `bg-surface`, `text-accent`, `text-ai`, etc.

### Component Patterns
- All shared UI components are in `components/UI.tsx` and use `forwardRef`
- Stage components receive `{ state: AppState; setState: Dispatch<SetStateAction<AppState>> }` as props
- AI-generated content uses the `ai` prop on Card or the `ai-border` / `ai-shimmer` CSS classes
- The `cn()` utility merges Tailwind classes safely — always use it for conditional classes

### State Management
- All app state lives in a single `AppState` object in `App.tsx` (no external state library)
- `projectId` tracks the current backend project; auto-created on first load
- Auto-save: changes are debounced (1s) and saved to backend via PATCH
- Stage components update state via `setState(prev => ({ ...prev, ... }))`

### AI Service Pattern
- Each AI service (concept_generator, card_generator) follows the same pattern:
  - Check for `ANTHROPIC_API_KEY` — if missing, return mock data
  - Call Claude API with structured system prompt requesting JSON output
  - Parse response, handle markdown-wrapped JSON
  - Fall back to mock data on any error
- This allows development without an API key while supporting real AI in production

### Design System
- **Dark mode default**, light mode via `.light-mode` class on `:root`
- Accent color: warm amber/gold (`#F59E0B`)
- AI indicator: purple gradient (`#8B5CF6` → `#D946EF`)
- Spacing: 8px grid
- Border radius: sm=4px, md=8px, lg=12px, xl=24px

### Keyboard Shortcuts
- `1`–`6`: Jump to pipeline stages
- `Alt+Left/Right`: Previous/next stage
- Shortcuts are disabled when focus is in input/textarea/select elements

## Development

```bash
cd app

```

### Backend
```bash
cd backend
pip install -e ".[dev]"
uvicorn app.main:app --reload --port 8000
```

```bash

CORS_ORIGINS=["http://localhost:3000"]
```

### Testing
```bash
cd backend
python -m pytest tests/ -v    # 33 tests
```

### API Endpoints

#### Projects (CRUD)
- `GET  /api/health` — health check
- `GET  /api/projects` — list projects
- `POST /api/projects` — create project (201)
- `GET  /api/projects/:id` — get full project with concepts, card types, cards
- `PATCH /api/projects/:id` — update project fields
- `DELETE /api/projects/:id` — delete project
- `POST /api/projects/:id/card-types` — add card type
- `POST /api/projects/:id/cards` — add card
- `PATCH /api/projects/:id/cards/:cardId` — update card

#### AI Generation
- `POST /api/generate/:id/concepts` — generate concept suggestions (Claude/mock)
- `POST /api/generate/:id/card-types` — generate card types (Claude/mock)
- `POST /api/generate/:id/cards` — generate card instances (Claude/mock)
- `POST /api/generate/:id/simulate` — run balance simulation (randomized engine)

#### Export
- `POST /api/export/:id/pnp-pdf` — generate Print-and-Play PDF

## Guidelines

- Frontend auto-creates a project on first load and persists state to backend
- When adding new AI services, follow the concept_generator.py pattern (Claude + mock fallback)
- Keep the `AppState` type in `types.ts` as the single source of truth for frontend
- Keep SQLAlchemy models in `backend/app/models/` as the single source of truth for persistence
- Don't add state management libraries — the current useState + auto-save approach is intentional
- Art/Layout stages still use placeholder content — these are next for implementation
- Backend uses SQLite for development; swap to Postgres via `DATABASE_URL` env var for production
